## 1. 问题诊断

### 1.1 用户观察

用户在 review 6 个 mockup 后指出：
> "这一个应用是一个视频通话的应用，是一个 AI 视频通话应用，但是这个页面做得好像并不像视频通话的应用"

### 1.2 根因分析

6 个 mockup 全部沿用了 XEngineer 当前的布局：**左右双栏 50/50（左摄像头 4:3 + 右聊天对话）+ 顶部 header + 底部 statusbar**。

这个布局的本质是**"开发者调试面板"**美学 —— 摄像头和聊天平等地并排展示，像是在调 WebSocket + VAD + TTS 链路的 demo 控制台，不像日常使用的视频通话应用。

### 1.3 与主流视频通话应用的对比

| 维度 | 主流视频通话（Zoom/FaceTime/Meet/微信） | 我的 6 个 mockup | 问题 |
|---|---|---|---|
| **视频画面** | 全屏铺满，是主角 | 左栏 4:3 小窗（占屏 ~25%） | 视频被边缘化 |
| **控制条** | 浮动底部，统一收纳 | 散落在左栏（录音/测试按钮各占一处） | 没有通话控制感 |
| **聊天/字幕** | 可隐藏侧拉，或字幕带浮在视频下方 | 永远占右半屏 | 文字喧宾夺主 |
| **状态信息** | 浮动顶部（通话时长/连接质量） | 固定底部 statusbar | 没有"通话中"的仪式感 |
| **挂断/静音/摄像头开关** | 必有红色挂断 + 麦克风 + 摄像头三件套 | 完全没有 | 缺视频通话的核心控件 |

主流视频通话应用的核心 UI 范式是：**全屏画面 + 浮动控制层 + 可隐藏的辅助层**。6 个 mockup 完全没遵循这个范式。

### 1.4 XEngineer 的特殊性

不能 100% 照搬主流视频通话，因为 XEngineer 有几个独特点：

1. **AI 没有自己的视频画面** —— 主流视频通话主角是"远端画面"，XEngineer 的主角是"AI 看到你的画面"（即你自己的摄像头预览）
2. **AI 回复既有语音（TTS）也有文字（流式）** —— 主流视频通话只有语音，所以 AI 文字回复需要可见空间
3. **VAD 是核心交互** —— 用户说话触发 AI 回应，需要可视化反馈
4. **摄像头截图是 AI 视觉输入** —— 不是给人看的"视频画面"，而是给 AI 看的"视觉样本"

**XEngineer 的本质** = **视频通话（用户画面是主角）** + **AI 字幕/聊天（AI 文字回复是次要层）** + **VAD 触发的对话流（不是按键通话）**

## 2. 4 个候选布局方向

### 候选布局 ①：全屏视频 + 浮动 AI 字幕（最像 FaceTime/Meet）

```
┌────────────────────────────────────────────────┐
│ ● 00:42  连接良好                  AI 正在思考 │  ← 顶部浮动状态
│                                                │
│                                                │
│           [你的画面全屏铺满]                    │
│           (VAD speaking 时画面边缘发光)         │
│                                                │
│                                                │
│  ┌──────────────────────────────────────────┐  │
│  │ AI: 是的，我能看到你。你戴着...▌         │  │  ← 浮动 AI 字幕条
│  └──────────────────────────────────────────┘  │  (半透明黑底，3秒淡出)
│                                                │
│       📹  🎤  🔇  📷  ⏹                       │  ← 底部浮动控制条
└────────────────────────────────────────────────┘
```

**特性**:
- 视频画面占 100% 视口，是绝对主角
- AI 流式回复以浮动字幕条形式显示在画面下方（半透明黑底）
- 字幕条 3 秒后自动淡出，鼠标 hover 时重新显示
- 历史对话需上滑呼出（从顶部下拉手势 / 顶部历史按钮）
- 顶部状态栏：通话时长 + 连接质量 + AI 状态
- 底部控制条：摄像头开关 / 麦克风 / 静音 / 截图 / 挂断

**优点**: 最像视频通话，画面是绝对主角，沉浸感最强
**缺点**: 历史对话被弱化（只在字幕条临时显示），需要额外手势呼出
**适合**: 用户主要靠语音对话，文字回复是辅助

### 候选布局 ②：大视频 + 右侧窄聊天栏（Zoom 风）

```
┌──────────────────────────────────┬─────────────┐
│ ● 00:42  连接良好               │ 对话历史    │
│                                  │ ┌─────────┐ │
│                                  │ │USER:你好│ │
│                                  │ └─────────┘ │
│      [你的画面占 70%]            │ ┌─────────┐ │
│                                  │ │AI:是... │ │
│      (VAD speaking 时边缘发光)   │ └─────────┘ │
│                                  │             │
│                                  │ ┌─────────┐ │
│                                  │ │AI 流式▌ │ │
│  📹  🎤  🔇  📷  ⏹              │ └─────────┘ │
└──────────────────────────────────┴─────────────┘
```

**特性**:
- 视频画面占 70%，右侧 30% 是聊天栏
- 聊天栏始终可见，显示历史对话 + 流式 AI 回复
- 顶部状态栏简化为浮动条
- 底部控制条浮动在视频区底部
- 移动端：聊天栏可折叠为底部抽屉

**优点**: 保留文字聊天可见性，视频仍是主角
**缺点**: 还是分屏（但比例反转为 7:3 而非 5:5），不如布局①沉浸
**适合**: 用户既要看画面也要看文字回复

### 候选布局 ③：全屏视频 + 右侧可折叠抽屉（最灵活）

```
默认状态（视频全屏）:
┌────────────────────────────────────────────────┐
│ ● 00:42  连接良好                       [💬]  │  ← 顶部状态 + 聊天按钮
│                                                │
│                                                │
│         [你的画面全屏铺满]                     │
│         (VAD speaking 时画面边缘发光)          │
│                                                │
│                                                │
│  ┌──────────────────────────┐                  │
│  │ AI: 是的，我能看到...▌   │  ← 浮动 AI 字幕  │
│  └──────────────────────────┘                  │
│                                                │
│       📹  🎤  🔇  📷  ⏹                       │  ← 底部控制条
└────────────────────────────────────────────────┘

点击 [💬] 后（右侧抽屉滑出）:
┌──────────────────────────────┬─────────────────┐
│ ● 00:42              [💬 ×] │ 对话历史        │
│                              │ ┌─────────────┐ │
│                              │ │USER: 你好   │ │
│                              │ └─────────────┘ │
│   [你的画面 60%]             │ ┌─────────────┐ │
│                              │ │AI: 是的,... │ │
│   (VAD speaking 时边缘发光)  │ └─────────────┘ │
│                              │                 │
│                              │ ┌─────────────┐ │
│  📹 🎤 🔇 📷 ⏹              │ │AI 流式 ▌    │ │
│                              │ └─────────────┘ │
└──────────────────────────────┴─────────────────┘
```

**特性**:
- 默认视频全屏，与布局①相同
- 区别：AI 字幕条不会自动消失，点击 [💬] 按钮呼出右侧聊天抽屉
- 抽屉滑出时视频区压缩到 60%，抽屉关闭时视频恢复全屏
- 抽屉内显示完整对话历史
- 移动端：抽屉变为全屏覆盖

**优点**: 默认像视频通话，需要时呼出聊天；用户对"是否看历史"有完全控制
**缺点**: 交互复杂度增加；新用户可能不知道 [💬] 按钮的作用
**适合**: 用户主要用语音，偶尔查看文字历史

### 候选布局 ④：视频主角 + 浮动全功能控制层（最激进）

```
┌────────────────────────────────────────────────┐
│ ● 00:42  AI 正在思考                  [X 挂断] │  ← 顶部浮动状态
│                                                │
│                                                │
│         [你的画面全屏铺满]                     │
│         (画面边缘随 VAD 状态变化)              │
│                                                │
│                                                │
│  ┌──AI 流式回复浮动卡片─────────────────────┐  │
│  │ 👂 你说: "你能看到我吗？"                │  │  ← 双层浮动卡片
│  │ 🤖 AI: 是的，我能看到你。你戴着...▌      │  │  (含用户语音识别 + AI 回复)
│  └─────────────────────────────────────────┘  │
│                                                │
│  ╭──控制条───────╮                            │
│  │ 📹 🎤 🔇 📷 ⏹ │  ← 浮动控制条（圆角胶囊）  │
│  ╰───────────────╯                            │
└────────────────────────────────────────────────┘
```

**特性**:
- 视频全屏铺满
- 顶部浮动状态栏（含挂断按钮，红色 X）
- 中下方浮动"对话卡片"：同时显示"用户语音识别"+"AI 流式回复"两层
- 底部浮动控制条用胶囊形圆角容器收纳
- 所有浮动元素半透明 + 模糊背景（backdrop-filter: blur）

**优点**: 所有信息浮在视频上，最像"AI 视频通话"，视觉层次丰富
**缺点**: 浮动卡片可能遮挡画面，需要精心设计透明度和位置；信息密度高
**适合**: 愿意为"AI 视频通话感"做出妥协

## 3. 调试能力依赖性研究结论（subagent-R1 报告）

> 完整报告: `docs/ui-research/debug-dependency-analysis.md`

### 3.1 核心结论

✅ **可以放心重做布局**。核心 pipeline 调试 100% 独立于 UI。

### 3.2 关键发现

1. **核心 pipeline 调试 100% 独立于 UI** — `tests/test_pipeline_e2e.py` 用 Python `websockets` 直连后端，覆盖完整 VAD→ASR→VLM→TTS 链路，与前端 0 耦合
2. **5 个测试脚本弱依赖 UI 文案** — `test_frontend_visual.sh` / `test_browser_e2e.sh` / `netlify_e2e_test.sh` 依赖按钮文案"录音"/"测试"、DOM 含"已连接"/"AI"字样等 12 个检查点
3. **前端状态管理与核心逻辑解耦** — React state 都是纯 UI 显示；核心链路（`audioPlayer.stop()` + `send vad_status` + `setShouldCapture`）在 `handleVADStateChange` 副作用中执行，不依赖 state
4. **后端独立可调试** — `GET /health` + Stub 模式 + Railway 日志
5. **无隐藏调试面板** — 前端无 `<details>` / `debugger` / dev 模式开关

### 3.3 重做布局时必须保留的

- `handleVADStateChange` 中的 3 个副作用调用（barge-in + ASR 启停 + 摄像头截图）
- `audioPlayer` 24kHz 全局单例
- `getUserMedia` 用户手势触发

### 3.4 重做布局时建议同步做的

- 给关键按钮加 `data-testid`，让测试脚本从文案依赖改为 testid 依赖（解决 12 个检查点问题）
- 抽 `useConversationOrchestrator` hook，把 VAD 链路从 UI 层抽离
- "发送测试消息"按钮：移除 UI 或移到"更多"菜单，但保留 WS 协议（`{type:'test', data}`）
- 提炼 `tests/ws_debug_client.py` 作为日常调试入口

### 3.5 调试能力对布局重设计的影响

**零阻塞**。布局可以完全重做，只需在 PR #1（基础层）或前置 PR 中同步加 `data-testid` 即可保持测试脚本不破坏。

## 4. 新布局下的 Signature 演化

> 用户提出的核心问题：**旧 signature 是基于旧布局设计的，新布局下有没有新的 signature？**

### 4.1 旧 Signature 的移植性分析

#### 示波器（方案 A）的 5 个 signature 元素

| 旧 signature 元素 | 旧位置 | 新布局下的演化 |
|---|---|---|
| 取景器十字线（4 角 L + 中心十字） | Camera 4:3 视频框 | ✅ **可移植** — 全屏视频的 4 角 L 形角标 + 中心十字，比例更大更显眼 |
| 32 柱示波器波形 | 录音按钮上方 | ⚠️ **需重新设计** — 不再是按钮上方的小波形，可做成：(a) 底部控制条上方的能量条 / (b) 全屏边缘的能量环（VAD speaking 时屏幕边缘光晕） |
| 数据读数条 | Header 右侧 | ✅ **可移植** — 顶部浮动状态栏的一部分 |
| 流式回复光标 | AI 气泡末尾 | ✅ **可移植** — 浮动字幕条/聊天栏内的光标 |
| 背景网格 | 整页底层 | ❌ **不再适用** — 全屏视频覆盖了背景，网格失去意义；可改为视频画面的"测量刻度"叠加层（仅在 VAD speaking 时显示） |

#### 暗房（方案 B）的 5 个 signature 元素

| 旧 signature 元素 | 旧位置 | 新布局下的演化 |
|---|---|---|
| 红色 ambient 光晕 | 整页左上角径向渐变 | ✅ **可移植** — 视频画面上叠加红色 ambient 光晕（混合模式：overlay），暗示"暗房显影"环境 |
| Camera 边框胶片穿孔 | Camera 视频框左右两侧 | ✅ **可移植** — 全屏视频左右两侧的胶片穿孔装饰（更细更密） |
| 流式回复显影动效 | AI 气泡文字模糊渐显 | ✅ **可移植** — 浮动字幕条内文字的显影动效 |
| "显影时间"进度指示 | StatusBar `DEV:12s` | ✅ **可移植** — 顶部浮动状态栏显示 `DEV 12s` 替代 "loading" |
| 印章式时间戳 | 每条消息底部 | ✅ **可移植** — 浮动字幕条右下角印章式时间戳 |

### 4.2 新布局产生的新 Signature 机会

新布局（视频通话风格）天然带来旧布局没有的视觉机会，这些是**新 signature 的候选**：

#### 新 Signature 候选 ①：VAD 画面边缘光晕（4 个布局通用）

**描述**: 用户说话时（VAD speaking 状态），全屏视频画面的四个边缘出现发光效果，像"AI 正在听"的视觉反馈。

**实现**:
- 用 CSS `box-shadow: inset 0 0 80px var(--brand-500)` 或 SVG 边框
- speaking 时 opacity 0 → 1 渐变，silent 时反向
- 配合 VAD 能量值动态调整光晕强度（音量越大光晕越亮）

**对示波器**: 光晕颜色用朱砂红，配合 32 柱波形（移到状态栏）形成"听觉+视觉"双反馈
**对暗房**: 光晕颜色用安全灯红，画面像"被暗房红灯照亮"

**这是新布局独有的 signature**，旧布局的 4:3 小窗做不出全屏光晕的沉浸感。

---

#### 新 Signature 候选 ②：通话计时器（4 个布局通用）

**描述**: 顶部浮动状态栏显示 `● 00:42` 格式的通话时长计时器，从用户首次开启摄像头/麦克风开始计时。

**实现**:
- React state `callDuration` + `useEffect` setInterval
- 显示格式：`MM:SS`（< 1 小时）/ `HH:MM:SS`（≥ 1 小时）
- 红色 `●` 圆点呼吸动画表示"通话进行中"

**对示波器**: 计时器用 JetBrains Mono 字体，配合数据读数条
**对暗房**: 计时器用 IBM Plex Mono，配合"显影时间"语义（"DEV 00:42"）

**这是视频通话应用的标志性元素**，旧布局完全缺失。

---

#### 新 Signature 候选 ③：AI 状态四态指示（4 个布局通用）

**描述**: 顶部状态栏显示 AI 当前状态，4 个状态切换：
- `IDLE` 空闲（等待用户说话）
- `LISTENING` 正在听（VAD speaking 中）
- `THINKING` 正在想（收到 ASR final 后、TTS 开始前）
- `SPEAKING` 正在说（TTS 播放中）

**实现**:
- 状态机：`vadStatus` + `isAIProcessing` + `audioPlayer` 播放状态 → 4 态
- 顶部状态栏显示当前态 + 颜色变化
- 每次状态切换有 200ms 过渡动效

**对示波器**: 用 mono 字体 + 数据读数条样式
**对暗房**: 用"显影"语义 — `IDLE` = 待显影 / `LISTENING` = 曝光中 / `THINKING` = 显影中 / `SPEAKING` = 已显影

**这是 AI 视频通话应用独有的 signature**，主流视频通话没有"AI 思考"概念。

---

#### 新 Signature 候选 ④：浮动控制条胶囊（布局 ①③④）

**描述**: 底部浮动控制条用胶囊形容器收纳所有按钮，半透明 + 模糊背景，hover 时整个胶囊微微上浮。

**实现**:
- 容器：`backdrop-filter: blur(20px)` + `background: rgba(0,0,0,0.4)` + `border-radius: 9999px`
- 按钮：圆形 48×48px，hover 时背景变亮
- 挂断按钮：红色 `#ef4444`，与其他按钮间距更大（视觉分隔）

**对示波器**: 控制条加细 1px 朱砂红边框
**对暗房**: 控制条加暖色调（`rgba(45, 24, 16, 0.6)`）+ 黄铜边框

**这是视频通话应用的标志性 UI**，旧布局完全没有"挂断"概念。

---

#### 新 Signature 候选 ⑤：用户语音识别字幕（布局 ④）

**描述**: 浮动卡片同时显示两层：
- 上层："👂 你说: " + 用户语音识别结果（来自 `asr_interim` / `asr_final`）
- 下层："🤖 AI: " + AI 流式回复（来自 `llm_chunk`）

让用户直观看到"我说了什么 → AI 怎么回的"完整对话流。

**实现**:
- 上层：`asr_interim` 实时显示（灰色，final 时变白）
- 下层：`llm_chunk` 流式显示 + 光标
- 两层之间有 hairline 分隔线

**对示波器**: 用 mono 字体 + 数据读数风格
**对暗房**: 上层用"曝光中"动效（模糊渐显），下层用"显影中"动效

**这是 AI 视频通话独有**，主流视频通话没有"自己说话的字幕"。

---

### 4.3 Signature 演化总结

| Signature 类型 | 示波器（A） | 暗房（B） |
|---|---|---|
| **可移植的旧 signature** | 取景器十字线 / 数据读数条 / 流式光标 | 红色 ambient 光晕 / 胶片穿孔 / 显影动效 / 印章时间戳 / DEV 进度 |
| **需重新设计的旧 signature** | 32 柱波形 → 状态栏能量条 或 全屏边缘能量环 | — |
| **不再适用的旧 signature** | 背景网格（被视频覆盖） | — |
| **新布局独有的新 signature** | VAD 边缘光晕（朱砂红）/ 通话计时器 / AI 四态指示 / 浮动控制条胶囊 / 用户语音字幕 | VAD 边缘光晕（安全灯红）/ 通话计时器（DEV 语义）/ AI 四态指示（显影语义）/ 浮动控制条胶囊（黄铜）/ 用户语音字幕（曝光+显影） |

**结论**: 新布局下，旧 signature 大部分可移植，同时新布局带来 5 个新 signature 机会。**新布局的 mockup 应该既体现旧 signature 的移植，又展示新 signature 的运用。**

## 5. 可切换文字回复的设计方案

### 5.1 用户诉求

> "他可以做成一个可切换的? 比如默认隐藏, 需要时呼出(前提是用户知道可以呼出文字回复)"

### 5.2 设计挑战

"默认隐藏 + 需要时呼出"的核心矛盾：
- 默认隐藏 → 用户可能不知道有文字回复 → 错过 AI 的关键信息
- 始终显示 → 喧宾夺主 → 不像视频通话应用

### 5.3 三个解决方案

#### 方案 A：智能显隐（推荐）

**规则**:
- 默认：AI 字幕条隐藏
- AI 开始流式回复时：字幕条自动浮现（300ms 渐显）+ 持续显示
- AI 回复结束 + 3 秒后：字幕条自动淡出
- 用户随时点击 [💬] 按钮：呼出完整对话历史抽屉

**"用户如何知道可呼出"**:
1. 首次进入应用时显示 2 秒的引导提示："点击 [💬] 查看对话历史"
2. [💬] 按钮在顶部状态栏始终可见，带未读小红点（有未查看的 AI 回复时）
3. AI 字幕条出现时，右下角小提示："点击查看完整对话"

**优点**: 智能平衡，AI 说话时文字必现，AI 不说话时画面干净
**缺点**: 实现复杂度中等

#### 方案 B：始终浮动 + 可隐藏

**规则**:
- 默认：AI 字幕条始终浮动显示（半透明）
- 用户点击 [👁 隐藏字幕] 按钮：字幕条隐藏
- 有新 AI 回复时：字幕条自动重现 + 持续 3 秒
- 用户点击 [💬] 按钮：呼出完整对话历史

**优点**: 实现简单，用户对"是否显示"有完全控制
**缺点**: 默认状态还是"有字幕条"，不如方案 A 干净

#### 方案 C：完全手动

**规则**:
- 默认：无字幕条，画面全屏
- 用户必须点击 [💬] 按钮才能看到任何文字
- 历史对话和流式回复都在 [💬] 抽屉里

**优点**: 最干净，最像视频通话
**缺点**: 用户可能错过 AI 回复（除非 AI 用语音播报），不适合听障用户

### 5.4 推荐

**推荐方案 A（智能显隐）**，因为：
1. 解决了"用户不知道可呼出"的问题（引导提示 + 未读小红点）
2. AI 说话时文字必现，保证信息传达
3. AI 不说话时画面干净，像视频通话
4. 完整历史通过 [💬] 抽屉访问

### 5.5 与 4 个布局的匹配

| 布局 | 推荐的可切换方案 |
|---|---|
| ① 全屏+字幕 | 方案 A（智能显隐）— 字幕条本来就会淡出 |
| ② 大视频+聊天栏 | 不需要切换 — 聊天栏始终可见 |
| ③ 全屏+抽屉 | 方案 A 或 C — 抽屉本身就是切换机制 |
| ④ 浮动全功能 | 方案 B（始终浮动 + 可隐藏）— 浮动卡片是 signature 一部分 |
